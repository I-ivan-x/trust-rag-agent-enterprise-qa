# PRINCIPLES — 项目宗旨与高维原则

适用范围：高于 Task Plan 的元层文档。Task Plan 规定做什么和怎么做，本文档规定当计划与现实冲突时按什么裁决。所有 scope review、计划修订、临时决策，以本文档为最终依据。

------

## 一、本项目真正交付的是什么

### P1. 交付物是"一个站得住的论断"，不是 44 个勾选框

项目的最终产品不是代码、不是链路，而是一句你能在任何追问下守住的话：

> "在这些条件下，这个系统比 naive RAG 更可信；在这些条件下，它仍然失败；我有证据。"

所有任务的价值按它对这句话的贡献度排序。当时间不够、必须二选一时：**一个更小但论证完整的系统，永远优于一个更大但论证有缺口的系统。** 砍模块不损害这句话，砍评测和失败分析直接摧毁它。

### P2. 诚实是技术策略，不只是道德要求

每一处超卖都是面试时的攻击面：被问穿一次，整个项目的可信度连坐归零——而这个项目的主题恰恰是"可信"。反过来，**有据可查的失败是最稀缺的展示材料**。能说"它能跑"的候选人遍地都是，能说"我知道它在哪里失效、为什么失效、我测过"的极少。因此：

- 任何指标不加限定条件不出现在任何文档里；
- 任何"提升"必须能指向产生它的那次 eval run 的 run_id；
- FAILURE_ANALYSIS 的篇幅和打磨投入不低于 EVALUATION_REPORT。

### P3. 项目的隐藏产品是你自己

面试官最终面的是人，不是 repo。这个项目同时在生产两个东西：一个系统，和一个能解释这个系统每处取舍的你。如果第二个产品缺货，第一个产品一文不值。这条原则直接导出 P7。

## 二、决策原则

### P4. 区分单向门与双向门，按不可逆性分配审慎度

- **单向门（错了极难回头）**：schema 设计、eval protocol、语料冻结、metadata 字段、ID 规则。这些决策的下游有冻结的数据和回填的 gold 标签，返工成本是乘法级的。单向门决策必须过夜——起草后隔天再确认，且 Owner 逐字审。
- **双向门（错了改就是）**：模块内部实现、prompt 措辞、阈值数值、目录组织。这些快速决策、快速试错，不开会、不纠结。

大部分进度灾难来自把单向门当双向门草率通过，把双向门当单向门反复纠结。

### P5. 计划是当前最优假设，不是合同

8 周里现实一定会教给计划不知道的事。原则是：**允许偏离，但偏离必须留痕。** repo 中维护 `docs/DECISIONS.md`，每条记录三行：日期、改了什么、为什么当时的计划是错的。带记录的偏离是工程判断力的证据；沉默的偏离是混乱；僵硬的遵守是另一种失败。Week 8 时这份文档会自动成为 PROJECT_REVIEW 的骨架和面试中"你踩过什么坑"的弹药库。

### P6. 抵抗单一数字的诱惑

可信性本质上是 trade-off（false refusal vs false answer、覆盖率 vs 准确率），**曲线和取舍论证才是产品，单点数字只是曲线上的一个截面**。任何把项目压缩成"准确率提升了 X%"的表述都是对项目的降维。简历上可以放数字，但每个数字背后必须有一条你能展开讲五分钟的 trade-off 故事。

## 三、AI 协作原则

### P7. 所有权不可外包：AI 执行，你裁决

本项目由 AI 大量代写代码和文档，这带来一个独有的致命风险：**项目完成了，但 Owner 答不出深度追问**——这在面试中比项目没做完更糟。对策是制度性的，不靠自觉：

- **白板测试**：每周验收时，Owner 必须能脱稿在白板上重画当周模块的数据流、并复述每个关键决策的理由。画不出来 = 该周不验收，先还"理解债"再继续。
- **决策权不下放**：AI 可以提供三个方案，选哪个、为什么选，必须是 Owner 的判断并记入 DECISIONS.md。"AI 建议的"在面试中不是一个可用的答案。
- **理解债与技术债同级管理**：看不懂的代码等于没有的代码。宁可让 Codex 重写一版更简单的，不留一段 Owner 讲不清的"高级实现"。

### P8. AI 的产出默认是初稿，关键路径产出默认有错

对 AI 生成的内容，审查强度按 P4 的不可逆性分配：单向门产出（schema、eval 数据、协议文档）逐字审；双向门产出抽查 + 测试兜底。特别地，**AI 写的 eval case 和 gold 标签是最危险的产出**——它们看起来最无害，错了却污染所有下游结论，且错误以沉默方式存在。

## 四、节奏原则

### P9. main 分支永远处于"今天就能给人看"的状态

不存在"最后再整理"——demo 会腐烂，README 会过期，能跑的东西三周不碰就不能跑了。机制：每周验收的最后一项固定是冷启动复现（按 README 从零跑通 quickstart + demo queries）。这条的成本是每周 20 分钟，省下的是 Week 7–8 的整周返工，以及任何时刻突然需要展示项目时的从容。

### P10. 失败案例即时捕获，不靠 Week 8 回忆

维护 `docs/FAILURE_JOURNAL.md`：开发和评测过程中**当场**记下每个有意思的失败（trace_id、现象、一句话猜测原因），30 秒一条，不求分析。Week 8 的 FAILURE_ANALYSIS 是对这本日志的整理升华,而不是凭记忆重建——重建出来的失败分析千篇一律,即时捕获的失败分析有细节和体温，面试官分得出来。

### P11. 最后 20% 的打磨决定 80% 的观感，必须结构性保护

评审者（面试官、招生官）接触项目的顺序是：README → 报告 → demo → 也许看代码。前三者都在 Week 7–8 的交付物里。Week 4 砍单机制保护的正是这个尾部——这不是行政流程，是本项目最重要的单项风控。任何"再加一个功能"的冲动，先问：它比把 EVALUATION_REPORT 写得更清楚优先级更高吗？答案几乎永远是否。

### P12. 不做清单是作品集的一部分

工程能力的最高展示形式不是"我会做什么"，而是"我判断什么不值得做、并说得出为什么"。Task Plan 中的 N-01 到 N-15 不是遗憾，是展品——面试时主动讲"我为什么没上 Elasticsearch / 没做多 Agent / 没做前端"，比讲任何已实现模块更能区分工程判断力。每次砍单后，把砍掉的内容和理由补进这份清单，而不是默默删除。

## 五、终局校验

### P13. 每周用同一个问题校准方向

> "如果项目今天被迫终止，我手上的东西能讲出什么？"

健康的答案应该每周都比上周完整一点——这要求价值是增量交付的，而不是押在 Week 8 的大爆炸上。如果连续两周这个问题的答案没有变化，说明在做的是基建自嗨，立即触发临时 scope review。

### P14. 终点不是 tag，是三个"能"

项目完成的真正定义：

1. **能复现**：一个陌生人按 README 在干净环境里 30 分钟内跑出 demo 和 eval；
2. **能讲**：Owner 能在 5 分钟、15 分钟、45 分钟三个时长版本下讲这个项目，且每个版本都以 P1 的那句论断收尾；
3. **能挨打**：对着 INTERVIEW_QA 里最刁的十个问题，每个都有不回避的答案——包括"这个项目最大的弱点是什么"。

三者缺一，tag 打了也不算完成。
